# 表分布、复制表和分布键

## Share-Nothing

OpenTenBase 属于分布式数据库语境。Share-Nothing 思路下，各 DN 通常持有自己的数据分片和计算资源，CN 负责协调查询入口和结果汇总。不要假设所有数据都在单个实例本地可见。

## 分布表

分布表将数据按规则分散到多个 DN，适合大表。查询可能只访问部分 DN，也可能访问多个 DN。分布表的 Join、聚合、排序和更新可能涉及跨节点协调。

## 复制表

复制表通常在多个节点保留副本，适合较小且变化较少的维表或配置表。复制表可以减少跨节点 Join，但副本维护会带来一致性和更新成本。

## 分布键

分布键决定一行数据落到哪个 DN。选择分布键时考虑：

- 是否高基数，避免大量行落到同一 DN。
- 是否经常作为查询过滤条件。
- 是否经常作为 Join key。
- 是否存在热点值或业务单边倾斜。
- 是否能减少跨 DN 事务或数据重分布。

## 数据倾斜

数据倾斜指部分 DN 存储或计算压力明显高于其他 DN。常见原因包括：

- 分布键基数低。
- 分布键存在热点值。
- 业务数据天然不均衡。
- 查询条件集中命中少数分片。

## Agent 判断影响

- 设计表时不能只问“主键是什么”，还要问“分布键是什么”。
- 排查性能时要关注 DN 间数据量和计算量是否均衡。
- 解释插件或 SQL 兼容性时，要考虑对象是否会在 CN/DN 两侧参与执行。
