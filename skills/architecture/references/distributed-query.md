# 分布式查询

## 为什么不同于单机 PostgreSQL

单机 PostgreSQL 中，SQL 通常在一个实例内规划和执行。OpenTenBase 中，用户连接 CN 后，CN 需要结合元数据、表分布、节点拓扑和查询条件，把一条 SQL 转化为分布式执行过程。

## SQL 基本流向

1. 用户连接 CN。
2. CN 解析 SQL。
3. CN 根据表分布和查询条件生成分布式计划。
4. CN 将可下推任务发送到相关 DN。
5. DN 执行本地数据访问。
6. CN 汇总、排序、聚合或合并结果后返回用户。

## Join 的分布式代价

Join 的代价取决于数据是否已经位于同一 DN：

- 如果两张分布表按相同分布键关联，可能在 DN 本地完成 Join。
- 如果一张是复制表，小表副本可让 DN 本地 Join，减少跨节点搬运。
- 如果 Join key 和分布键不匹配，可能需要广播、重分布或 CN 汇总，代价更高。

## 对插件和 SQL 的影响

- 插件函数是否能在 DN 安全执行，影响函数下推。
- SQL 中的函数、类型、操作符和聚合不能只看 CN 上是否存在，还要考虑 DN 执行环境。
- 性能排查要关注是否发生数据重分布、广播或 CN 汇总瓶颈。

## Agent 判断影响

- 不要只回答“SQL 语法正确”，还要判断它是否适合分布式执行。
- 遇到慢 SQL，优先询问表分布、分布键、Join key、过滤条件和数据倾斜。
- 遇到插件适配，优先确认插件对象是否需要在 CN 和 DN 都可见。
