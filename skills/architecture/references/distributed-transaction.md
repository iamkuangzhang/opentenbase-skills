# 分布式事务

## 基本概念

OpenTenBase 是分布式数据库。涉及多个 DN 的写入、读写一致性和故障恢复，不能简单等同于单机 PostgreSQL 的本地事务。

- 单节点事务：只涉及一个 DN，协调成本较低。
- 跨节点事务：涉及多个 DN，需要全局协调。
- 只读查询：也可能需要一致快照或全局可见性语义。
- 节点异常：可能影响提交、回滚或恢复。

## GTM 与两阶段提交

GTM 和相关事务机制用于维护全局事务语义。跨 DN 写操作通常需要协调多个参与者，常见分布式数据库会使用两阶段提交一类协议保证提交一致性。

两阶段提交的风险包括：

- 参与节点故障导致提交阻塞。
- 协调者故障导致恢复复杂。
- 网络或节点异常时需要额外状态判断。

## Agent 判断影响

- 不要把单机事务成本直接套用到跨 DN 写入。
- 遇到批量更新、跨分布键更新、多表写入，要提示可能存在分布式事务成本。
- 解释故障或恢复时，要区分 CN 可连接、DN 可达、GTM/事务协调是否正常。
- 不要为了验证事务机制直接写入生产库。
